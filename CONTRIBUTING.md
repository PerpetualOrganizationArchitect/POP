Thank you for your interest in contributing to the Perpetual Organization Protocol. POP is a production protocol with real organizations and real treasuries on it; the rules below exist to keep it that way. Contributions are accepted under the project's [AGPL-3.0 license](LICENSE).

If anything below is unclear, ask in [Discord](https://discord.gg/9SD6u4QjTt) or open a discussion issue. Clarifications are themselves valuable contributions.

## Before You Start

1. Read [`docs/POP_OVERVIEW.md`](docs/POP_OVERVIEW.md) for the protocol's philosophy and architecture.
2. Read [`CLAUDE.md`](CLAUDE.md), a dense rules-of-the-road document. It is the single best summary of the project's conventions.
3. Skim [`SWITCHABLE_BEACON.md`](SWITCHABLE_BEACON.md) if your change touches upgradeability or beacon plumbing.
4. Browse open issues and PRs to make sure your work doesn't duplicate something in flight. Look for the `good first issue` and `help wanted` labels.
5. **For non-trivial changes, open an issue first** describing the problem and your proposed approach. We'd rather align on direction up front than ask you to redo a PR.
6. For security-sensitive findings, follow the [responsible-disclosure process](SECURITY.md). Do **not** open a public issue for a vulnerability.

## Local Setup

```bash
# 1. Fork the repo on GitHub, then clone your fork
git clone --recurse-submodules git@github.com:<you>/POP.git
cd POP

# 2. (If you forgot --recurse-submodules)
git submodule update --init --recursive

# 3. Configure environment
cp .env.example .env
# Edit .env. Only DEPLOYER_PRIVATE_KEY is needed if you intend to deploy.

# 4. Verify the toolchain
forge --version

# 5. Build and test
forge build
forge test -vvv
```

Submodules: `forge-std`, `openzeppelin-contracts`, `openzeppelin-contracts-upgradeable`, `hats-protocol`, `solady`. Keep them at the pinned commits; do not bump them as a side effect of unrelated work.

**Do not run `foundryup`.** Foundry is pre-installed in CI; updating the toolchain locally is fine for your own machine, but never as part of a PR.

## Hard Rules

These mirror [`CLAUDE.md`](CLAUDE.md) and are enforced by CI, code review, or both. Reviewers will block PRs that violate them.

| Rule | Why |
|------|-----|
| `forge fmt` must pass before commit/PR | CI rejects unformatted Solidity. |
| Custom errors only (**no `require()` with string messages**) | Gas + consistency. Pattern: `if (cond) revert CustomError();`. |
| ERC-7201 namespaced storage (**no `__gap` arrays**) | Append-only `struct Layout` keyed by `keccak256("poa.<module>.storage")`. |
| Every upgradeable constructor calls `_disableInitializers()` | Initialization happens via proxy `initialize(...)` only. |
| Hats Protocol for all role-based access (**no OpenZeppelin `AccessControl`**) | Hats is the protocol-wide identity model. |
| Named imports only: `import {Foo} from "./Foo.sol";` | Avoid pollution from wildcard imports. |
| Match existing pragma when editing a file | The codebase spans `^0.8.17`–`^0.8.30`; bumping a pragma can break dependents. |
| Do not edit `upgrades/` | Auto-generated by CI for upgrade-safety validation. |
| Do not skip CI hooks (`--no-verify`) or signing flags | Investigate and fix the underlying failure. |

**Optimizer is OFF in the default profile.** This is deliberate; see [`foundry.toml`](foundry.toml). A Solidity IR codegen bug causes `vm.roll()` to silently produce wrong results when the optimizer is enabled. Use `FOUNDRY_PROFILE=production` only for real deployments, never for tests.

## Code Style

- **NatSpec.** Every external/public function needs at least an `@notice`. Use `@param`/`@return` when intent is non-obvious.
- **Errors.** Define custom errors at the top of the contract, grouped under a `/* ─────────── Errors ─────────── */` comment per the existing pattern.
- **ERC-7201.** Use the `_layout()` private-pure-assembly accessor pattern shown in [`src/Executor.sol`](src/Executor.sol). Do not hand-roll the assembly differently.
- **Hat checks.** Always go through [`src/libs/HatManager.sol`](src/libs/HatManager.sol) (`hasAnyHat`, `setHatInArray`, etc.) rather than raw `hats.isWearerOfHat()` calls.
- **Address validation.** Use `ValidationLib.requireNonZeroAddress(addr)` at boundary entrypoints.
- **No comments narrating WHAT.** Names should already make that clear. Comments are for non-obvious WHY (constraints, invariants, workarounds for specific bugs).
- **No author/issue references in code comments** like "added by X for ticket Y"; those belong in the PR description.

## Commit and PR Conventions

**Commit titles.** Use [Conventional Commits](https://www.conventionalcommits.org) with the merging PR number:

```
feat: add cashout relay (#123)
fix: prevent reentry in PaymentManager.claimDistribution (#142)
refactor: extract PaymasterGraceLib (#138)
docs: rewrite CLAUDE.md with architecture and gotchas (#137)
test: add invariant suite for HybridVoting (#150)
chore: bump forge-std to v1.9.6 (#155)
```

The PR number is appended once the PR is open and you know what it'll be (squash-merge replaces the title automatically; if you're rebase-merging, edit before merge).

**PR titles.** Same format as the eventual squash commit. Keep them under ~70 characters; put detail in the description.

**PR descriptions.** State the *why*, not just the *what*. Include:
- A 1–2 sentence summary of the problem and your approach.
- Any storage-layout impact (and a link to the regenerated `upgrades/` files in the PR diff).
- Whether it requires a paired PR in `subgraph-pop` or `Poa-frontend`.
- A short test plan: which tests you ran, which scenarios you exercised manually.

**Branch names.** Anything sensible. The team prefers `<your-handle>/<concrete-feature>` (e.g., `alice/cashout-relay`).

## Testing Requirements

- Tests live in [`test/`](test). Each contract gets a paired suite at `test/<ContractName>.t.sol`.
- Run `forge test -vvv` locally before pushing; that's the verbosity CI uses.
- New code paths require new tests. Bug fixes require a regression test that fails before the fix.
- Use `vm.roll()` for block advancement (necessary for voting and timelock tests). The optimizer-off default is what makes that reliable; never re-enable the optimizer in the test profile.
- Naming: `test` + descriptive scenario, e.g. `testMintOnlyAuthorized`, `testPauseAndUnpause`. For revert cases, append the error: `testTransfer_revertsTransfersDisabled`.
- Internal test helpers: prefix with `_`, e.g. `_createSimpleOrg()`.
- Mocks under [`test/mocks/`](test/mocks): minimal interface implementations only. Do not pull in heavyweight stand-ins.
- The three `.t.sol.skip` PaymasterHub files require an EntryPoint fork; they are not run in CI. Rename to `.t.sol` to enable locally if you're working in that area.

Coverage report (CI also runs this):

```bash
forge coverage --report summary
```

CI's coverage filter is `src/`, so coverage reflects production code only.

## Storage Layout Changes

POP uses ERC-7201 namespaced storage with strict append-only semantics. The CI workflow validates every PR's storage layout against [`upgrades/baseline/`](upgrades/baseline) using [`script/upgrades/ValidateUpgrade.s.sol`](script/upgrades/ValidateUpgrade.s.sol). The build fails on any reorder, removal, or type change of an existing field.

If your change *intentionally* updates a `Layout`:

1. **Append only.** Add new fields at the end of the struct.
2. **Never remove or reorder.** Even unused fields must stay.
3. **Never change a field's type.** A `uint128` cannot become a `uint256` without a slot collision.
4. **The HybridVoting v2 namespace is shared across three libraries.** `HybridVotingCore`, `HybridVotingConfig`, and `HybridVotingProposals` all read/write the same slot (`keccak256("poa.hybridvoting.v2.storage")`). Changes must be coordinated across all three.
5. **Do not edit `upgrades/` by hand.** CI regenerates these files. The `upgrades/baseline/` snapshot will be updated by maintainers when an upgrade is released.

Surface storage-layout impact prominently in your PR description so reviewers can validate the diff.

## Coordinating with the Subgraph and Frontend

POP is a multi-repo system. Two changes typically require paired PRs in sister repos:

**Event signature changes** affect [poa-box/subgraph-pop](https://github.com/poa-box/subgraph-pop). If you rename, remove, or change the parameter list of an indexed event (`TaskCompleted`, `ProposalCreated`, `Voted`, `Transfer`, `QuickJoined`, etc.), the subgraph mappings must be updated in lockstep:

1. Open the contracts PR in this repo.
2. Open a paired PR in `poa-box/subgraph-pop` updating `pop-subgraph/schema.graphql` and the relevant mappings.
3. Cross-link the two PRs.

If a tracked event's signature lands in production without a subgraph update, the frontend will silently lose data for that entity.

**ABI changes** affect [poa-box/Poa-frontend](https://github.com/poa-box/Poa-frontend), which keeps copies of contract ABIs under `poa-app/abi/`. Substantive ABI changes (new external functions, parameter changes) usually warrant a paired frontend PR; pure internal refactors don't.

## PR Checklist

Paste this into your PR description and tick what applies:

```markdown
- [ ] `forge fmt` passes
- [ ] `forge build` passes (default profile)
- [ ] `FOUNDRY_PROFILE=production forge build` passes
- [ ] `forge test -vvv` passes locally
- [ ] New behavior has test coverage; bug fixes have regression tests
- [ ] If event signatures changed: paired PR opened in `poa-box/subgraph-pop` (link: …)
- [ ] If ABIs changed substantively: paired PR opened in `poa-box/Poa-frontend` (link: …)
- [ ] If `Layout` changed: append-only; storage-layout impact called out above
- [ ] NatSpec `@notice` on every new public/external function
- [ ] Custom errors used (no `require` strings)
- [ ] No new `__gap` arrays; no OpenZeppelin `AccessControl`
- [ ] Named imports only
- [ ] Existing pragma preserved on edited files
- [ ] Conventional-commit title with `(#PR)` suffix
- [ ] Slither runs clean locally (or known-acceptable findings noted in description)
```

## CI Gates Explained

Defined in [`.github/workflows/ci.yml`](.github/workflows/ci.yml), delegating to a reusable workflow in `BreadchainCoop/etherform`. On every PR and push to `main`:

| Step | What it does | How to satisfy |
|------|--------------|----------------|
| `check-formatting` | Runs `forge fmt --check` | Run `forge fmt` locally before pushing. |
| `forge build` | Compiles in default + production profiles | Fix any compiler errors. |
| `forge test` | Runs all `.t.sol` files at `-vvv` | Fix failing tests; add coverage for new code. |
| `run-coverage` | Generates coverage filtered to `src/` | Add tests if coverage drops on new code. |
| `run-slither` | Static analysis | Fails on `high` severity. `arbitrary-send-eth` is excluded (see [`slither.config.json`](slither.config.json)). For other findings, justify in the PR or fix. |
| `run-upgrade-safety` | Validates layouts against `upgrades/baseline/` via `script/upgrades/ValidateUpgrade.s.sol` | Storage changes must be append-only. Maintainers update the baseline at release time. |
| `flatten-contracts` | Generates flattened source for verification | No action needed; auto-runs. |

CI does *not* deploy on PR or main (`deploy-on-pr: false`, `deploy-on-main: false`). Deploys are manual operations driven by maintainers using the scripts in [`script/`](script).

## Adding a New Module

A typical "add a new org-level module" change looks like:

1. **Implementation contract** at `src/MyModule.sol`:
   - Inherits the appropriate OpenZeppelin upgradeable bases (`Initializable`, plus `OwnableUpgradeable`, `PausableUpgradeable`, `ReentrancyGuardUpgradeable` as needed).
   - Constructor calls `_disableInitializers()`.
   - All state in a single `struct Layout` accessed via `_layout()` at slot `keccak256("poa.mymodule.storage")`.
   - External entrypoints validate inputs (`ValidationLib.requireNonZeroAddress(...)`) and check Hats permissions (`HatManager.hasAnyHat(...)`).
   - Custom errors defined at the top; NatSpec on every external function.
2. **Tests** at `test/MyModule.t.sol`:
   - Cover happy path, every revert path, edge cases, and any reentrancy concern.
   - Use mocks from `test/mocks/` where possible.
3. **Wiring**:
   - If it's an org-level module, add construction to the appropriate factory in [`src/factories/`](src/factories).
   - Register the implementation type in the deployment script under [`script/deploy/`](script/deploy).
   - Add the new contract to the relevant org config JSONs under [`script/config/`](script/config) if it's user-configurable.
4. **Beacon**: the factory will deploy a `SwitchableBeacon` for the module. Confirm Mirror mode is the default unless governance reasons require otherwise.
5. **Subgraph**: if the module emits indexable events, add them to the subgraph schema (paired PR in `poa-box/subgraph-pop`).
6. **Docs**: drop a short `docs/MY_MODULE.md` if the module is non-trivial. Link it from [`README.md`](README.md#documentation-index).
